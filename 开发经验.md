# TinyNote 日期翻页与数据库交互 BUG 修改报告

日期：2026-06-17

## 1. 背景

TinyNote 在 Windows MVP 初版中出现了一组连续相关的稳定性问题。最初表现为日期快速翻页后，任务完成、添加、修改、删除、清空以及设置保存都可能失效，需要多次点击才可能生效。后续多轮修复后，设置保存和任务完成有所缓解，但添加任务仍然失败，并最终暴露出数据库启动级错误：

```text
TinyNote 启动失败
error returned from database: (code: 1) cannot commit - no transaction is active
```

这次问题不是单点 UI bug，而是由日期导航、任务加载、routine 自动生成、sync_log 写入、SQLite 写队列和 Tauri SQL 插件连接池共同触发的一条竞态链。

## 2. 问题演进

### 2.1 第一阶段：日期翻页后所有操作变慢或失效

用户连续点击日期右箭头后，出现这些现象：

- 当前选中日期可能不在可见日期范围内。
- 点击完成任务大概率无效。
- 添加新任务大概率无效。
- 修改任务、删除任务、清空已完成任务大概率无效。
- 设置保存也会受影响。

当时主要怀疑点集中在 `taskStore` 的加载竞态。后续确认存在以下问题：

- `navigateDate()` 会提前递增 `latestTaskLoadId`，但实际 `loadTasks()` 被防抖延后执行。
- 在这段延迟窗口内，用户的添加/修改/完成等操作也会触发加载，结果被未来的导航加载判定为过期。
- 过期加载的提前返回路径没有总是恢复 `isLoading`。
- 多次快速翻页后，UI 容易进入“看似还活着，但操作结果被旧加载覆盖或丢弃”的状态。

### 2.2 第二阶段：设置和完成任务改善，但添加仍失败

后续修复将日期翻页改成即时更新，并让任务完成走乐观路径后：

- 设置保存明显改善。
- 完成任务明显改善。
- 但添加任务仍然失败或卡住。

这说明第一阶段修复只解决了前端 store 竞态的一部分，数据库写入链路仍有更底层问题。

### 2.3 第三阶段：出现 `cannot commit - no transaction is active`

最终出现启动失败：

```text
error returned from database: (code: 1) cannot commit - no transaction is active
```

这个错误把问题指向了数据库事务层。检查后发现，前端代码通过 Tauri SQL 插件手动执行：

```sql
BEGIN IMMEDIATE
...
COMMIT
```

但 Tauri SQL 插件的 JS API 只有 `execute/select/close`，其文档和实现说明它面向 Rust 侧 connection pool。多次 `db.execute()` 是多次 invoke，不保证使用同一条 SQLite 连接。于是可能出现：

1. `BEGIN IMMEDIATE` 在连接 A 上执行。
2. 中间写入可能在连接 A 或其他连接上执行。
3. `COMMIT` 在连接 B 上执行。
4. 连接 B 没有活动事务，于是 SQLite 返回 `cannot commit - no transaction is active`。

这就是本轮最底层根因。

## 3. 根因总结

本次 BUG 链路可以分成三层。

### 3.1 Store 层竞态

`taskStore` 中日期翻页、任务加载和任务写操作共享同一组状态标记：

- `latestTaskLoadId`
- `pendingNavigationLoad`
- `isLoading`
- `visibleDates`
- `visibleStartDate`
- `selectedDate`

早期实现中，导航加载、用户任务操作加载、routine 生成后的刷新加载互相覆盖，导致用户操作结果被旧请求丢弃或覆盖。

### 3.2 后台任务和同步日志阻塞主操作

任务操作并不只是写一条任务记录，它还会触发：

- SQLite 写操作。
- sync_log 写入。
- 任务重新加载。
- routine 自动生成。
- routine 生成后再次刷新任务列表。

当用户快速翻页时，routine 自动生成会不断补齐可见日期范围，sync_log 也会排队写入。普通任务写入被这些后台操作拖住，用户就会感觉“点了没反应”。

### 3.3 Tauri SQL 插件连接池与手写事务不兼容

这是最终导致启动失败和添加任务失败的核心根因。

前端手写事务依赖“BEGIN、写入、COMMIT 必须在同一连接上执行”。但 Tauri SQL 插件通过 connection pool 执行命令，JS 侧无法保证多次 `execute` 绑定同一连接。

因此，前端侧跨多次 invoke 手写 SQL 事务是不安全的。

## 4. 已完成修改

### 4.1 日期翻页即时响应

修改位置：

- `desktop/src/stores/taskStore.ts`

主要变化：

- `navigateDate()` 只负责即时计算并更新日期窗口。
- 翻页后的任务加载使用 120ms 防抖，只执行最后一次加载。
- 连续点击右箭头时，下一次计算基于最新 `selectedDate/visibleDates`。
- 旧加载晚返回时不会覆盖新窗口。

效果：

- 当前选中日期保持在可见日期范围内。
- 快速翻页不再让 UI 状态停在旧日期。

### 4.2 任务操作改成优先路径

修改位置：

- `desktop/src/stores/taskStore.ts`
- `desktop/src/pages/MainPage.tsx`

主要变化：

- `addTask/updateTask/completeTask/archiveTask/restoreTask/deleteTask` 开始时会取消待执行导航加载。
- 任务完成、归档、恢复、删除采用乐观更新。
- 添加任务在 service 返回后直接合并进当前 store，不再总是等待完整 reload。
- 清空已完成任务从 `Promise.all` 并发删除改成顺序删除，避免写队列洪峰。

效果：

- 任务完成和删除能立即反馈。
- 快速翻页后的任务操作不会再被旧导航加载覆盖。

### 4.3 routine 自动生成降级为后台补齐

修改位置：

- `desktop/src/stores/taskStore.ts`
- `desktop/src/services/routineService.ts`
- `desktop/src/repositories/routineRepository.ts`

主要变化：

- `loadTasks()` 先加载并渲染已有任务，立即结束 `isLoading`。
- daily routine 生成改为后台执行。
- 后台生成完成后，只有当前窗口仍然有效时才刷新任务。
- 旧 routine 生成结果晚返回时不会覆盖用户刚完成/删除/修改的任务状态。

效果：

- routine 自动补齐不再阻塞日期导航和用户任务操作。

### 4.4 sync_log 改为后台低优先级写入

修改位置：

- `desktop/src/services/syncLogService.ts`
- `desktop/src/repositories/db.ts`

主要变化：

- 新增后台写入口 `executeBackgroundWrite()`。
- `writeSyncLog()` 默认非阻塞，失败只记录日志。
- 用户主操作不再等待 sync_log 成功。

效果：

- sync_log 不再拖慢任务添加、完成、删除和设置保存。
- sync_log 仍保留未来同步能力，但不影响 MVP 本地操作成功。

### 4.5 SQLite 写队列拆分前台和后台优先级

修改位置：

- `desktop/src/repositories/db.ts`

主要变化：

- 写队列拆成 foreground 和 background。
- 普通任务写入、设置写入走 foreground。
- sync_log 和 routine 自动补齐走 background。
- 队列执行时优先清空 foreground，再执行 background。

效果：

- 用户主动操作不会排在大量后台 sync_log/routine 写入之后。

### 4.6 移除前端手写 SQL 事务

修改位置：

- `desktop/src/repositories/db.ts`

主要变化：

- 移除 `BEGIN IMMEDIATE`。
- 移除 `COMMIT`。
- 移除 `ROLLBACK`。
- 保留 JS 层串行批处理，避免不同写操作在批处理过程中插入。

当前注释：

```ts
// The Tauri SQL plugin executes commands through a pool, so manual BEGIN/COMMIT
// across separate invoke calls can land on different connections. Keep these
// batches serialized in JS instead of opening SQL transactions here.
```

效果：

- 修复 `cannot commit - no transaction is active`。
- 避免启动时默认设置初始化和 routine 批量写触发连接池事务错误。

### 4.7 routine 生成插入顺序修复

修改位置：

- `desktop/src/repositories/routineRepository.ts`

早期风险：

- 先插入 `routine_instances`。
- 再插入 `tasks`。
- 如果 task 插入失败，可能留下 orphan routine instance。

当前逻辑：

- 先按条件插入 task。
- task 插入成功后，再写 `routine_instances`。
- 如果发现已有相同 `routine_id + instance_date`，跳过 task 插入。

效果：

- stale generation 不会生成重复任务。
- 避免先占位 instance 但 task 没有写入的坏状态。

### 4.8 添加任务失败后表单恢复

修改位置：

- `desktop/src/components/TaskInput.tsx`

主要变化：

- `handleSubmit()` 增加 `try/catch/finally`。
- 添加失败后记录错误并恢复 `isSaving=false`。
- 不再因为一次 DB 异常导致添加按钮永久 disabled。

效果：

- 添加失败后，用户可以继续重试。

### 4.9 版本升级

版本从 `0.1.8` 升到 `0.1.9`。

涉及文件：

- `desktop/package.json`
- `desktop/package-lock.json`
- `desktop/src-tauri/tauri.conf.json`

## 7. 已成功解决的问题

- 日期快速翻页后 selectedDate 不在 visibleDates 中的问题。
- 日期快速翻页后任务完成大概率无效的问题。
- 日期快速翻页后设置保存被拖慢的问题。
- 大量后台 routine/sync_log 写入拖慢用户主动操作的问题。
- `isLoading` 被旧加载卡住的问题。
- `database is locked` 频繁出现时缺少重试和写队列协调的问题。
- `cannot commit - no transaction is active` 启动失败问题。
- 添加任务失败后输入框按钮一直 disabled 的问题。
- routine stale generation 重复生成或先写 instance 的数据一致性风险。

## 8. 未完全解决或仍需注意的问题

### 8.1 当前不再使用真正 SQLite 原子事务

为了修复 Tauri SQL 连接池下的 `cannot commit - no transaction is active`，当前移除了前端手写 SQL 事务。

现在的保证是：

- JS 层串行批处理。
- 同一批 callback 内不会被其他 JS 写队列任务插入。
- 避免 connection pool 跨连接 COMMIT 错误。

现在不保证的是：

- 多条 SQL 在 SQLite 层的严格原子提交。

MVP 本地单窗口场景可以接受，但长期更好的方案是把需要事务的批量写迁到 Rust command，由 Rust 侧持有同一连接并执行真正事务。

### 8.2 routine 生成仍有极小一致性窗口

当前顺序是：

1. 插入 task。
2. 再插入 routine_instance。

这避免了“instance 存在但 task 不存在”的更坏情况。但如果应用刚好在 task 插入后、routine_instance 插入前崩溃，可能留下一个 routine task 没有 instance 登记。

后续可通过 Rust 侧事务彻底解决。

### 8.3 添加失败暂时只写 console

`TaskInput` 已经不会卡死，但失败时还没有用户可见提示。

后续建议：

- 底部 toast。
- 输入框错误态。
- 简短错误文案，例如“保存失败，请重试”。

### 8.4 尚未做安装版长时间压力脚本

目前自动化测试覆盖了关键竞态，但没有做真实安装版的脚本化压力测试，例如：

- 连续点击右箭头 100 次。
- 同时添加 50 条任务。
- 开启 daily routine 后反复翻页。
- 关闭重启后核对数据一致性。

建议后续补 Playwright 或 Rust/JS 层压力脚本。

## 9. 关键经验

### 9.1 不要在前端侧跨多次 invoke 手写数据库事务

只要底层是连接池，多次 `execute()` 就不能假设同一连接。事务必须在能够绑定连接的层实现。

对于 Tauri：

- 简单单条写：前端 SQL plugin 可以接受。
- 多条强一致写：应放到 Rust command。
- 批量但可弱一致：可用 JS 串行队列。

### 9.2 UI 竞态和 DB 竞态要分层处理

这次一开始只修 UI 竞态，确实改善了完成任务和设置保存，但没有解决添加失败。原因是底层 DB 事务仍有独立 bug。

经验：

- UI store 竞态解决“状态覆盖”。
- 写队列解决“操作排队和优先级”。
- 数据库事务模型解决“持久化正确性”。

三层不能混为一个问题。

### 9.3 后台任务不能阻塞用户主动操作

routine 自动生成和 sync_log 都是后台能力，不应该成为本地 MVP 的主操作前置条件。

本项目中应遵循：

- 用户主动任务写入：高优先级。
- 设置保存：高优先级。
- routine 自动补齐：低优先级。
- sync_log：低优先级，默认非阻塞。

### 9.4 失败态必须解除 UI 锁

`TaskInput` 的问题提醒我们，所有 `isSaving`、`isLoading`、`disabled` 状态都必须有 `finally` 或等价回收路径。

否则一次失败就会变成“用户需要多点几次”或“看起来整个功能坏了”。

### 9.5 测试必须覆盖晚返回和旧请求覆盖新状态

这类 bug 的关键不是单个请求是否成功，而是：

- 旧请求晚返回。
- 后台任务晚返回。
- 用户操作夹在导航防抖窗口中。
- loading 在过期路径中是否恢复。

后续所有类似状态管理都应包含这类测试。

## 10. 后续建议

### 10.1 中短期

- 在添加任务失败时显示用户可见错误提示。
- 给 taskStore 增加操作错误状态，便于 UI 反馈。
- 增加安装版手工验收清单：
  - 开启 daily routine。
  - 连续右翻 20 次。
  - 立即添加任务。
  - 立即完成任务。
  - 修改标题。
  - 删除任务。
  - 清空已完成。
  - 重启应用后核对数据。

### 10.2 中长期

- 将批量写和 routine 生成迁移到 Rust command。
- Rust 侧使用真正 SQLite transaction。
- 前端 repository 只调用 command，不直接组合多条 SQL。
- 为 DB 写入增加统一错误分类：
  - busy/locked。
  - constraint conflict。
  - migration mismatch。
  - connection/permission failure。

## 11. 当前结论

本轮核心问题已经修复并通过自动化验证。当前 0.1.9 版本已经解决：

- 日期翻页导致任务操作失效。
- 后台 routine/sync_log 拖慢主操作。
- 添加任务失败后 UI 卡死。
- Tauri SQL 连接池下手写事务导致的 `cannot commit - no transaction is active`。

仍需注意的是，当前前端 DB 层采用 JS 串行批处理而非 SQLite 原生事务。对于 MVP 是合理折中；如果后续继续增加复杂数据写入，应尽快迁移到 Rust 侧事务。

## 12. 透明圆角窗口方形边框修复

### 12.1 问题表现

应用主体已经是圆角便笺样式，但圆角外侧的透明矩形窗口区域内仍能看到一圈方形边框或矩形阴影。视觉上像是圆角便笺被放在一个透明但带边框的矩形容器里。

### 12.2 根因判断

TinyNote 的 Tauri 主窗口配置为：

```json
{
  "decorations": false,
  "transparent": true,
  "shadow": true
}
```

在 Windows 透明无边框窗口中，`shadow: true` 可能由系统按原始矩形窗口边界绘制阴影或边缘，而不是按 Web 内容的 CSS 圆角绘制。这会导致 CSS 内部的 `.app-shell` 虽然已经有 `border-radius: 18px`，但窗口外层仍出现矩形视觉痕迹。

此外，透明窗口需要确保 `html`、`body`、`#root` 都显式透明，并且主壳层对 backdrop/filter/背景做圆角裁剪，否则 WebView 合成时也可能出现矩形残留。

### 12.3 修改内容

修改 `desktop/src-tauri/tauri.conf.json`：

```json
"shadow": false
```

关闭 Tauri/系统原生矩形阴影，避免它覆盖 CSS 圆角效果。

修改 `desktop/src/styles/global.css`：

```css
html,
body,
#root {
  background: transparent;
}

.app-shell {
  clip-path: inset(0 round 18px);
}

.startup-error {
  clip-path: inset(0 round 18px);
}
```

### 12.5 后续注意

如果之后需要恢复阴影，不建议重新打开 Tauri 原生 `shadow`。更稳的方式是：

- 保持窗口原生 `shadow: false`。
- 在 Web 内容内部预留透明 padding。
- 使用 CSS 自绘圆角阴影或 `filter: drop-shadow(...)`。

这样阴影可以跟随圆角，而不会回到系统矩形窗口边界。
